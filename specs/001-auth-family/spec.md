# Feature 规约: 用户认证与默认家庭初始化

**Feature 分支**: `001-auth-family`

**创建日期**: 2026-07-06

**状态**: Draft

**输入**: 用户描述 (基于 `docs/PRD.md` 与 `docs/MVP.md`,经交互确认为首个 MVP feature)

## 概述

家庭记账系统的入口功能: 用户通过邮箱+密码注册或登录,系统为每个新用户自动创建一个「默认家庭」并在其下初始化一个代表用户本人的成员记录。这是 MVP 的根功能,后续所有 feature (账户、分类、交易、Dashboard) 都依赖它建立的「家庭 + 成员」上下文。

本 feature 仅覆盖单用户、单家庭、单成员的最小闭环;多成员、邀请、权限、第三方登录、密码找回均不在范围。

## Clarifications

### Session 2026-07-06

- Q: 默认登录态在多少天无操作后必须重新认证? → A: 30 天空闲超时 + 滑动续期 (活跃用户每次访问顺延,不重新登录)。
- Q: 密码强度策略采用哪种规则集? → A: 长度 ≥ 8 + 弱口令黑名单 (NIST 800-63B),不强制复杂度。
- Q: 认证相关事件必须记录到何种粒度? → A: 仅记关键安全事件 (注册成功、登录成功、登录失败、锁定触发、登出),含邮箱+时间戳+结果,不含密码或会话凭证;不记 IP/UA (延后 V2)。
- Q: 用户在锁定窗口内尝试登录时,系统应返回什么? → A: 显式提示 "已锁定,请 N 分钟后重试";其余情况仍返回通用错误。
- Q: 注册端点是否需要、以及如何做反垃圾保护? → A: 全局 IP 限流 (同 IP 每小时最多 10 次注册),不引入 CAPTCHA。

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 新用户注册并自动建立家庭 (Priority: P1)

新用户在注册页输入邮箱和密码,提交后系统创建用户、默认家庭、初始成员三项记录,并立即建立登录态,跳转到 Dashboard。

**为何此优先级**: 这是 MVP 的入口;没有用户和家庭,后续所有功能都无上下文。注册成功率直接决定产品漏斗顶端。

**独立测试**: 注册成功后,即使后续 feature (账户/交易) 尚未实现,也必须能验证「用户存在、家庭存在、成员存在、登录态有效」四项。

**Acceptance Scenarios**:

1. **Given** 该邮箱从未注册, **When** 用户提交有效邮箱+符合强度要求的密码, **Then** 用户、默认家庭、初始成员三条记录被创建,用户进入登录态。
2. **Given** 邮箱格式无效 (如缺少 `@`), **When** 用户提交, **Then** 返回字段级错误,不创建任何记录。
3. **Given** 该邮箱已注册, **When** 用户再次提交注册, **Then** 返回友好错误 (不暴露"邮箱已存在"以外的信息),原有记录不变。
4. **Given** 两次密码输入不一致, **When** 用户提交, **Then** 在前端拦截,不发往后端。
5. **Given** 密码强度不足 (少于 8 位或常见弱口令), **When** 用户提交, **Then** 返回强度要求说明,不创建记录。

---

### User Story 2 - 老用户登录 (Priority: P1)

已注册用户在登录页输入邮箱和密码,验证通过后建立登录态并跳转到 Dashboard。

**为何此优先级**: 没有登录就无法回到已记账的数据;这是日常使用的入口,与 US1 同等关键。

**独立测试**: 给定一个已存在的用户 (可通过 US1 准备),登录成功后必须能取到该用户的家庭和成员上下文。

**Acceptance Scenarios**:

1. **Given** 邮箱与密码均正确, **When** 用户提交登录, **Then** 进入登录态,返回当前用户、家庭、成员标识。
2. **Given** 邮箱不存在, **When** 用户提交, **Then** 返回通用错误 ("邮箱或密码错误"),不区分两种情况。
3. **Given** 邮箱存在但密码错误, **When** 用户提交, **Then** 返回同样的通用错误。
4. **Given** 同一邮箱连续失败 5 次, **When** 第 6 次尝试, **Then** 临时锁定该邮箱 5 分钟 (防止撞库),且第 6 次响应 MUST 显示"已锁定,请 5 分钟后重试"。
5. **Given** 锁定窗口已结束, **When** 用户输入正确密码登录, **Then** 登录成功,失败计数清零,响应恢复正常。
6. **Given** 用户已处于登录态, **When** 再次访问登录页, **Then** 直接跳转 Dashboard,不重复登录。

---

### User Story 3 - 登出 (Priority: P2)

已登录用户主动登出,系统清除登录态,返回登录页。

**为何此优先级**: P2 而非 P1 —— 登出不阻断核心记账流程,但属于基本卫生需求,缺少会让用户不得不清浏览器。

**独立测试**: 登出后再次访问受保护页面应被拒绝。

**Acceptance Scenarios**:

1. **Given** 用户处于登录态, **When** 用户点击登出, **Then** 当前会话失效,前端跳转登录页。
2. **Given** 用户未登录 (或会话已过期), **When** 调用登出, **Then** 操作幂等成功,无错误。

---

### User Story 4 - 查询当前会话 (Priority: P2)

前端在页面加载时调用一个端点,获取「当前是谁、属于哪个家庭、是哪个成员」三项标识,用于后续所有业务请求的上下文构造。

**为何此优先级**: P2 —— 不是用户直接看见的功能,但前端需要它来渲染任何业务页面 (Dashboard、流水)。它在 P1 之后立即被需要。

**独立测试**: 登录后立即调用此端点,应返回与登录响应一致的三个标识。

**Acceptance Scenarios**:

1. **Given** 用户已登录, **When** 前端查询当前会话, **Then** 返回用户 ID、家庭 ID、成员 ID,以及邮箱 (用于显示)。
2. **Given** 未登录或会话已过期, **When** 前端查询当前会话, **Then** 返回明确的"未登录"状态码,引导跳转登录页。
3. **Given** 用户刚刚登出, **When** 前端用旧会话查询, **Then** 同样返回"未登录"。

---

### Edge Cases

- 注册过程中网络中断,后端已部分写入 → MUST 通过事务保证用户/家庭/成员三条记录原子写入,要么全成功要么全失败。
- 用户邮箱大小写不一致 (如 `A@x.com` 与 `a@x.com`) → MUST 在存储时规范化为小写,确保唯一性。
- 邮箱首尾空白 → MUST 自动 trim。
- 密码包含 Unicode 字符 → MUST 接受并按字节计数强度,不因编码问题拒绝。
- 同一邮箱被并发注册 → 仅一个成功,另一个收到"已存在"错误。
- 同一 IP 在一小时内批量注册 → 第 11 个起返回 429,不写入任何记录;NAT 后多用户共用 IP 的极端场景下,真实用户可能被波及,UX 上 MUST 提示"稍后再试"而非"操作失败"。
- 登录时密码错误 N 次后正确 → 锁定窗口结束后正确登录必须成功,且锁定窗口内 MUST 给出"已锁定"的明确提示 (而非通用错误)。
- 用户长时间不操作后会话过期 → MUST 友好提示并保留用户已输入的页面 (如未提交的表单数据)。
- 默认家庭的名称在多语言环境下 → MVP 固定为中文「我的家庭」,国际化在 V2。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 允许新用户使用邮箱 + 密码注册。
- **FR-002**: 系统 MUST 校验邮箱格式 (RFC 5322 简化版),并拒绝重复邮箱。
- **FR-003**: 系统 MUST 强制密码最小长度 8 位,并拒绝常见弱口令 (如 `12345678`、`password`、`qwerty123`)。系统 MUST NOT 强制复杂度规则 (大小写混合、数字、符号) —— 依据 NIST 800-63B。弱口令黑名单 MUST 至少包含 OWASP top 1000 条。
- **FR-004**: 注册成功时,系统 MUST 在单次事务中原子地创建: 一个 User、一个默认 Family、一个代表该用户的 Member。
- **FR-005**: 默认家庭 MUST 拥有固定名称「我的家庭」,且 `Member.familyId` 指向它。
- **FR-006**: 系统 MUST 允许已注册用户用邮箱 + 密码登录。
- **FR-007**: 登录失败时,系统 MUST 返回通用错误信息,不得区分"邮箱不存在"与"密码错误"。
- **FR-008**: 系统 MUST 在登录成功后建立登录态,该状态 MUST 在浏览器刷新后保持有效。会话 MUST 设有 30 天空闲超时 (idle timeout);每次用户主动访问受保护端点时 MUST 滑动续期 30 天,即活跃用户不需要重新登录。绝对寿命上限不做强制 (允许长期活跃用户持续续期)。
- **FR-009**: 系统 MUST 对同一邮箱的连续失败登录进行节流: 5 次失败后锁定该邮箱 5 分钟。锁定窗口内,系统 MUST 返回明确的"已锁定,请 N 分钟后重试"提示 (而非通用错误);锁定窗口结束后,系统 MUST 恢复返回通用错误并重置失败计数。
- **FR-010**: 系统 MUST 允许已登录用户主动登出,登出 MUST 使当前会话立即失效。
- **FR-011**: 系统 MUST 提供"查询当前会话"端点,返回当前用户 ID、家庭 ID、成员 ID、邮箱。
- **FR-012**: 未认证访问受保护端点时,系统 MUST 返回明确的"未认证"状态。
- **FR-013**: 系统 MUST 使用加盐慢哈希存储密码,严禁明文或弱哈希 (MD5/SHA1)。
- **FR-014**: 系统 MUST 在日志与响应体中屏蔽密码字段,绝不返回密码或其哈希。
- **FR-015**: 系统 MUST 将邮箱在存储前规范化为小写并去除首尾空白。
- **FR-016**: 系统 MUST 记录以下认证事件: 注册成功、登录成功、登录失败、锁定触发、登出。每条记录 MUST 包含: 事件类型、邮箱 (或不存在则标记)、UTC 时间戳、结果 (成功/失败)。MUST NOT 包含密码或会话凭证字段。
- **FR-017**: 系统 MUST 提供按邮箱查询其最近 30 天认证事件的能力 (用于事件响应,如用户报告可疑活动)。
- **FR-018**: 系统 MUST 对注册端点做基于来源 IP 的全局限流: 同一 IP 每小时最多成功注册 10 个账户。超限时 MUST 返回 429 状态,且 MUST NOT 创建任何用户/家庭/成员记录。
- **FR-019**: 系统 MUST NOT 在注册端点引入 CAPTCHA 或第三方行为验证 (延后到 V2 评估)。

### Key Entities *(include if feature involves data)*

- **User**: 用户登录身份。属性: id、email (规范化后唯一)、password hash、创建时间。一个 User 在 MVP 中只对应一个 Family。
- **Family**: 记账聚合根 (依据 `docs/DOMAIN.md`)。属性: id、name (MVP 固定「我的家庭」)、创建时间。一个 Family 在 MVP 中只有一个 Member。
- **Member**: 家庭下的成员,代表"谁记了这笔账"。属性: id、familyId、display name (MVP 默认取 User 邮箱的本地部分)、创建时间。
- **Session**: 登录态。属性: 当前 User、Family、Member 标识、过期时间。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 新用户从打开注册页到完成注册并跳转 Dashboard,在 60 秒内完成 (P50)。
- **SC-002**: 老用户在已知邮箱+密码前提下,从打开登录页到 Dashboard 出现首个内容元素,在 5 秒内完成 (P95, 移动端 4G 网络)。
- **SC-003**: 99% 的有效凭证登录尝试,在单次提交内成功。
- **SC-004**: 任何 API 响应、日志、错误信息中均不出现密码原文或哈希值 (静态扫描 + 抓包验证)。
- **SC-005**: 任意已注册用户 MUST 可被验证为恰好关联 1 个 Family 与 1 个 Member —— 无孤儿、无重复。
- **SC-006**: 注册接口对同一邮箱的并发请求,至多成功一次,其余收到"已存在"错误。
- **SC-007**: 非法登录尝试 (错误密码) 5 次后,第 6 次即使在锁定窗口内输入正确密码,也必须被拒绝。
- **SC-008**: 用户主动登出后,旧会话凭证访问受保护端点 MUST 被拒绝。
- **SC-009**: 持续活跃的用户 (每 ≤30 天内有访问) MUST 在 90 天观察窗口内不需要重新登录;连续 31 天无访问的用户 MUST 在第 31 天起被强制重新认证。
- **SC-010**: 任一已注册邮箱的"最近 30 天认证事件"查询 MUST 在 5 秒内返回完整列表 (P95),且列表中任一字段均不得为密码或会话凭证。
- **SC-011**: 同一 IP 在 1 小时窗口内提交第 11 次注册请求时,MUST 收到 429 拒绝,且前 10 次之外的请求 MUST NOT 在数据库中产生任何写入。

## Assumptions

- 用户唯一标识为邮箱 (而非用户名);邮箱在存储前规范化为小写。
- 邮箱验证 (验证邮件、找回密码) 不在 MVP 范围,V2 实现。
- 密码找回不在 MVP 范围,运营层面通过直连 DB 处理紧急情况。
- 一个用户在 MVP 中只持有一个有效会话;并发会话限制暂不强制 (即用户可在多个设备同时登录)。会话默认 30 天空闲超时 + 滑动续期 (见 FR-008、SC-009)。
- 默认家庭名称固定为「我的家庭」;自定义家庭名称在 V2。
- 默认成员的显示名取邮箱本地部分 (@ 之前);自定义昵称在 V2。
- 锁定策略: 5 次失败后 5 分钟,基于邮箱而非 IP (因为家庭场景多在 NAT 后)。
- 注册接口不需要图形验证码;撞库防护通过 FR-009 的锁定机制 + FR-018 的 IP 限流兜底。
