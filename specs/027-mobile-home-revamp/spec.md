# Feature Specification: 手机端首页及相关页面重做

**Feature Branch**: `027-mobile-home-revamp`

**Created**: 2026-07-14

**Status**: Draft

**Input**: User description: "按照《个人记账软件手机端首页设计》(2026-07-14) 及配套 5 屏线稿(首页/明细/记一笔/统计/设置)重做手机端首页及相关页面。设计文档含转账、预算、资产概览、退款冲减等能力,这些在当前宪章 v3.1.0 原则一/三中属 MVP 范围外,本次已确认解锁全部范围(需先修宪章 v3.2.0 MAJOR)。"

## 宪章前置依赖 (BLOCKING)

本次 spec 的功能**直接违反宪章 v3.1.0 原则一(MVP 范围)与原则三(领域驱动)**,因为设计文档要求:

- **转账(transfer)交易类型** —— 原则一明确禁止:"未经 PRD 修订,禁止新增范围外功能 —— **转账**、预算、AI、OCR、导入导出、投资、多币种等。"
- **预算(Budget)能力** —— 原则三明确:"新表(`Asset`、`Debt`、`Budget`、`Investment`)在路线图解锁前属于范围外。"
- **资产概览(Asset/Debt 聚合)** —— 原则三同上。
- **退款冲减原支出分类** —— 原则一(退款不在 MVP 明确范围内)。

宪章治理章节规定:"任何新表、新 tRPC 路由、新 server action 必须能追溯至 MVP 中的某一项","冲突必须通过修订下位文档解决,而非忽略本宪章","修订须满足:书面提案 + 迁移计划 + SemVer 升级"。

**因此本 spec 视 `constitution.md` 从 v3.1.0 → v3.2.0 (MAJOR) 修订为**硬性前置依赖**。修订要点见本 spec §"宪法修订提案";修订落地前,User Story 4(转账)、5(预算)、6(资产)、退款相关 FR 均不可进入 plan/tasks 阶段。**

## User Scenarios & Testing *(mandatory)*

优先级按"先解锁前置、再首页核心、再配套页"的顺序排。User Story 1 是修宪章本身(它解锁后面所有 MVP 外功能),独立可测;User Story 2-3 是首页 UI/数据(在 MVP 范围内,可独立交付);User Story 4-6 依赖 US1 落地。

### User Story 1 - 解锁 MVP 外范围(宪章 v3.2.0 修订) (Priority: P0)

把宪章从 v3.1.0 升到 v3.2.0 (MAJOR),在原则一(MVP 范围)和原则三(领域驱动)中**解锁转账、预算、资产聚合、退款**四类能力,并附针对现有代码(027 之前仅 `income/expense` 两类交易、无 budget 表、无 asset 聚合)的迁移说明。修订满足宪章治理章节对 MAJOR 修订的三项要求:书面提案 + 迁移计划 + 版本号。

**Why this priority**: 宪章是"仓库内最高权威文档"。US4(转账)、US5(预算)、US6(资产)以及退款相关 FR 全部违反 v3.1.0 原则一/三。不先修宪章就实现这些功能,等于绕过治理流程,代码审查与 `/speckit-plan` 的"宪章检查"环节会直接否决。P0 = 阻塞一切后续。

**Independent Test**: 修订后的 `constitution.md` 头部版本号变为 3.2.0;原则一不再把"转账/预算"列为禁止项;原则三不再把 `Budget`/`Asset`/`Debt` 表列为"范围外";治理章节最后修订日期更新。可用一次 diff 检视独立验证。

**Acceptance Scenarios**:

1. **Given** 宪章 v3.1.0 原则一含"禁止新增范围外功能 —— 转账、预算…"字样, **When** 应用 v3.2.0 修订, **Then** 该字样被移除或改写为"以下能力自 027 解锁:转账、预算、资产聚合、退款",并在原则一保留"投资/OCR/导入导出/AI 仍属范围外"的硬约束。
2. **Given** 宪章 v3.1.0 原则三含"`Asset`、`Debt`、`Budget` 在路线图解锁前属于范围外", **When** 应用 v3.2.0 修订, **Then** 该句改写,允许 `Budget`/`Asset`/`Debt` 作为 `Family` 聚合内的新实体,但保留"跨聚合引用必须用 ID"等不变量约束。
3. **Given** 修订完成, **When** 执行一次 `/speckit-plan` 的宪章检查, **Then** 转账/预算/资产相关的 FR 不再被标记为"违反原则一/三"。

### User Story 2 - 首页布局与数据重做(MVP 范围内) (Priority: P1)

把手机端首页(`/dashboard`)按设计文档 §3.1-3.2 与线稿"首页"屏重做。重做包括:顶部导航(账本名 + 年月箭头切换 + 消息入口)、本月收支卡(主数字=**本月支出**,辅以收入/结余)、支出分类 top4 横向进度条、收支趋势(本月每日 + 上月同期对比)、最近账单(含左滑删除+撤销)、底部导航改为 4 入口 + 中央上凸圆形记账 FAB。资产/预算区块由 US5/US6 注入,本 US 先留位。

**Why this priority**: 首页是最高频入口,当前实现(026-dashboard-ui-refinement)主数字是"结余"而非"支出",与设计文档§3.1-2 直接冲突;且 026 刚下架的 Top2 分类在设计文档里被要求恢复为 Top4。这些是用户最直接感知的修正,P1。

**Independent Test**: 在只有收支两类交易(US4 未落地)的情况下也能完整测试首页布局、主数字语义、分类条、趋势、最近账单左滑、导航 FAB,并交付价值。US4/5/6 缺席时资产/预算区显示引导态,不阻塞。

**Acceptance Scenarios**:

1. **Given** 用户本月有收支记录, **When** 打开首页, **Then** 收支卡主数字是"本月支出"(非结余),收入/结余作为辅助行显示在其下方。
2. **Given** 用户切换月份(箭头点击或左右滑动), **When** 月份变化, **Then** 收支、分类、趋势、最近账单区块同步刷新到所选月;未来月份不可选。
3. **Given** 用户有 ≥4 个支出分类, **When** 查看"支出分类"区, **Then** 默认展示支出最高的 4 个分类,横向进度条 + 名称 + 金额。
4. **Given** 隐私模式开启, **When** 查看首页及趋势图, **Then** 所有金额(含趋势图 Y 轴刻度、数据点 Tooltip、点击金额)统一显示为遮蔽符,趋势图保留形状,分类名称保留但金额遮蔽。
5. **Given** 用户左滑最近账单某条, **When** 滑动到阈值, **Then** 露出删除;点击删除后短时间内(可配置,默认 5 秒)出现撤销入口;撤销则恢复该条。

### User Story 3 - 配套页(明细/统计)对齐线稿 (Priority: P2)

把"明细"页(`/transactions`)与"统计"页(`/reports`,线稿文案为"统计")按线稿对齐:明细页按日分组 + 类型筛选含**转账** tab(依赖 US4);统计页加 月/年 周期 toggle、日均/较上期对比卡、消费数据三宫格(最高支出日/最大单笔/支出次数),复用现有环形图。

**Why this priority**: 配套页是首页的下钻目标(点分类进明细带筛选、点趋势进统计)。首页 US2 先行可独立交付,但下钻目标缺转会断链;P2 紧随首页。

**Independent Test**: 明细页可在转账(US4)未落地时先用"全部/支出/收入"三 tab 测试(转账 tab 隐藏或空态);统计页三宫格与对比卡可纯前端聚合,独立测。

**Acceptance Scenarios**:

1. **Given** 明细页已加载, **When** 切换类型筛选, **Then** "全部/支出/收入/(转账)"按 US4 是否落地动态出现;按日分组且组头显示当日支出/收入/转账小计。
2. **Given** 统计页已加载且选"月"周期, **When** 查看摘要区, **Then** 显示本月支出总额 + 日均支出 + 较上月对比;趋势区显示"12 日最高 ¥580"类峰值徽标。
3. **Given** 统计页切换"年"周期, **When** 切换生效, **Then** 标签变为"全年支出/月均支出",峰值徽标变为"X 月最高",分类占比按全年聚合。

### User Story 4 - 转账(transfer)交易类型 (Priority: P2)

引入第三种交易类型 `transfer`(转账):一笔转账关联**转出账户 + 转入账户**,金额相同符号相反地影响两账户余额,**不计入收入或支出统计**。记一笔表单与明细/统计聚合规则同步支持。

**Why this priority**: 设计文档 §3.1-4(记一笔弹支出/收入/**转账**)、§5(转账不计入收支)、§8(验收:转账不影响收支总额)均要求。P2:依赖 US1(解锁)+ 受 US2 首页 FAB 弹层约束,但可独立测(单测聚合规则)。

**Independent Test**: 即使首页 UI(US2)未改,后端聚合规则也可用 tRPC `createCaller` + 集成测试独立验证:插入一笔 transfer,断言它不进 `monthIncome`/`monthExpense` 的 SUM。

**Acceptance Scenarios**:

1. **Given** 记一笔选"转账"模式并填好转出/转入账户 + 金额, **When** 保存, **Then** 系统生成一笔转账记录,转出账户余额减少、转入账户余额增加,金额相等。
2. **Given** 某月有转账记录, **When** 计算该月收支, **Then** 转账金额不计入收入或支出,结余 = 收入 - 支出(转账被排除)。
3. **Given** 明细页某日含转账, **When** 查看该日组头小计, **Then** 转账金额单独显示(如"转账 ¥500"),不混入支出/收入小计。

### User Story 5 - 预算进度 (Priority: P3)

在首页"本月收支卡"下方增加预算进度区:已用金额、剩余预算、使用百分比、时间进度 vs 预算进度对照;四态(正常/接近超支 80% 阈值/已超支 100%/未设置),未设置时显示"设置预算"轻量入口。

**Why this priority**: 设计文档 §3.1-3 + §4.3 + §8(验收:80%/100% 状态)。依赖 US1(解锁 Budget 表)。P3:首页核心(US2)已独立可交付,预算是增强层。

**Independent Test**: 预算查询/写入 procedure 可用 `createCaller` 独立测;前端四态可组件级测(传入不同 usage% 断言渲染)。

**Acceptance Scenarios**:

1. **Given** 用户设了本月预算且已用 63%, **When** 查看首页预算区, **Then** 显示"已用 ¥3,680 / 剩余 ¥2,119",进度条 63%,提示"本月已过 45% · 预算使用 63%"。
2. **Given** 预算使用达到 80%, **When** 查看, **Then** 显示"接近超支"预警态。
3. **Given** 预算使用超过 100%, **When** 查看, **Then** 显示"已超支"及超支金额。
4. **Given** 用户未设预算, **When** 查看首页, **Then** 预算区显示"设置预算"轻量入口,不显示空进度条。

### User Story 6 - 资产概览 (Priority: P3)

在首页下滑区底部增加资产概览:未添加账户时显示"添加第一个账户"引导;添加后显示净资产、总资产、总负债。点击进入账户与资产页。

**Why this priority**: 设计文档 §3.2-8 + §8(验收:未添加账户时显示引导)。依赖 US1(解锁 Asset/Debt 语义)。用现有 `Account` 表 + 新增 `type`(asset/debt)字段聚合(见 Clarifications Q1),不新增表。P3。

**Independent Test**: 资产聚合 procedure 独立可测;前端空/有态组件级可测。

**Acceptance Scenarios**:

1. **Given** 用户有 4 个账户(含 1 个负债账户), **When** 查看资产概览, **Then** 显示净资产/总资产/总负债三项,净资产 = 总资产 - 总负债。
2. **Given** 用户未添加任何账户, **When** 查看首页, **Then** 资产区显示"添加第一个账户"引导,点击进入账户管理。

### Edge Cases

- 320px 宽屏(最小支持宽度)下首页所有模块不出现横向滚动、内容重叠或文字截断(设计文档 §8 验收末条)。
- 新用户零交易状态下,首页保留顶部导航 + 简化收支概览,主按钮"记第一笔",不展示空白趋势图或空白分类排行(§6.1)。
- 部分配置态:有账单无预算/无账户、有账户无账单等组合,各模块分别显示对应引导态而非空卡片(§6.2)。
- 单个模块加载失败时,其余首页模块仍可用,失败模块显示简短说明 + 重试入口(§6.3)。
- 隐私模式开启时,趋势图必须隐藏 Y 轴金额刻度、数据点 Tooltip 金额、点击金额(当前实现仅遮蔽 `[data-amount]` 文本节点,**未遮蔽 recharts YAxis tick**,这是已知缺口)。
- 退款:采用"反向支出"(`expense` 类型、正金额、归原分类),不新增实体;同一分类下正负抵消,聚合时该分类净支出下降。退款作为反向支出参与左滑删除/编辑/撤销等与普通支出一致的交互。
- 转账的转出/转入账户为同一账户时,系统应拒绝(避免自转)。
- 预算预警阈值 80% 是否用户可配,还是硬编码 80%(§4.3 写"第一版设为 80%")—— 默认硬编码,可配置留 V2。
- 月/年统计的"较上期"对比基期定义(上月/去年同月)—— 默认月模式对比上月、年模式对比去年。

## Requirements *(mandatory)*

### Functional Requirements

#### 首页布局(US2,MVP 范围内)

- **FR-001**: 首页本月收支卡 MUST 以"本月支出"为主数字(大字),收入与结余作为辅助行;当前实现以"结余"为主数字的,必须反转。
- **FR-002**: 首页顶部导航 MUST 显示账本名称、当前年月、上月/下月切换入口;切换支持箭头点击与左右滑动两种手势,未来月份不可选。
- **FR-003**: 切换月份后,首页收支、(预算)、分类、趋势各模块 MUST 同步刷新到所选月份。**最近账单区独立于月份** —— 始终显示全局最新 3-5 条(跨月),不随月份切换变化(与设计 §3.2-7"最近账单"语义 + 现有 026 `recentTransactions` 实现一致;§4.1"同步刷新"仅约束受月份影响的模块)。
- **FR-004**: 首页"支出分类"MUST 默认展示支出最高的 4 个分类,横向进度条 + 分类名 + 金额;点击分类进入带月份+分类筛选的明细列表。当前 026 实现已下架 Top2,本 FR 要求恢复并扩展为 Top4。
- **FR-005**: 首页"收支趋势"MUST 默认展示本月每日支出趋势,并提供与上月同期的简短对比(如"较上月 -8%");点击数据点可查看对应日期金额(隐私模式下不可见金额)。
- **FR-006**: 首页"最近账单"MUST 展示最近 3-5 条,每条含分类/备注/账户/金额/时间;点击进入编辑,左滑可删除,删除后短时间内允许撤销。
- **FR-007**: 底部导航 MUST 为 4 个一级页面(首页/明细/统计/我的),记账按钮独立悬浮在导航中央上方(上凸圆形 FAB),不作为第 5 个普通导航项;点击 FAB **直接打开记一笔表单(默认支出模式)**,类型切换在表单内 mode-row(收入/支出/转账),不插二级选项弹层(符合极速录入目标;与线稿"记一笔"屏一致)。

#### 隐私模式补强(US2,MVP 范围内)

- **FR-008**: 隐私模式开启时,MUST 遮蔽应用内所有金额,包括趋势图的 Y 轴刻度、数据点 Tooltip 金额、点击金额;趋势图保留形状。当前实现仅遮蔽挂 `[data-amount]` 的文本节点,未覆盖 recharts 的 YAxis tick,必须补齐。
- **FR-009**: 隐私模式 MUST 应用到整个应用并跨重启保持(当前已实现,本次保持不变量)。

#### 配套页(US3)

- **FR-010**: 明细页 MUST 按日分组展示账单,组头显示当日支出/收入/(转账)小计;类型筛选含 全部/支出/收入(US4 落地后追加 转账)。
- **FR-011**: 统计页 MUST 提供月/年周期 toggle;摘要区显示(本期支出总额 / 日均或月均 / 较上期对比);趋势区显示峰值徽标;消费数据三宫格显示最高支出日/最大单笔/支出次数。

#### 转账(US4,依赖 US1 解锁)

- **FR-012**: 系统 MUST 支持第三种交易类型 `transfer`,一笔转账关联转出账户与转入账户,金额在两账户间等额移动。
- **FR-013**: 转账金额 MUST NOT 计入月度收入或支出统计,结余计算公式不变(收入 - 支出,转账排除)。
- **FR-014**: 记一笔表单选"转账"模式时,MUST 要求填转出账户与转入账户;两者为同一账户时 MUST 拒绝。
- **FR-015**: 转出账户余额 MUST 减少,转入账户余额 MUST 增加,金额相等。

#### 预算(US5,依赖 US1 解锁)

- **FR-016**: 系统 MUST 支持为账本设置**月度**预算金额(单位与交易一致,分),键为 (familyId, year, month),唯一;不支持年预算(统计页"年"周期不消费预算数据)。
- **FR-017**: 首页预算进度区 MUST 展示已用金额、剩余预算、使用百分比、时间进度 vs 预算进度对照。
- **FR-018**: 预算状态 MUST 区分四态:正常(<80%)/接近超支(≥80% 且 <100%)/已超支(≥100%)/未设置;接近超支与已超支须给出明确视觉提示与超支金额。
- **FR-019**: 未设置预算时,预算区 MUST 显示"设置预算"轻量入口,不显示空进度条。

#### 资产概览(US6,依赖 US1 解锁)

- **FR-020**: 首页资产概览区 MUST 显示净资产、总资产、总负债三项,净资产 = 总资产 - 总负债。总资产/总负债按账户 `type`(`asset`/`debt`)分组聚合(见 Key Entities)。
- **FR-021**: 未添加账户时,资产区 MUST 显示"添加第一个账户"引导,点击进入账户管理。

#### 退款(依赖 US1 解锁)

- **FR-022**: 系统 MUST 用"反向支出"实现退款:`type` 为 `expense`、`amount` 为正(抵消原负支出),归属原支出分类。退款使该分类净支出下降,**不计入普通收入**。不新增实体,不强制关联原交易。语义上等价于"在该支出分类下记一笔金额为正的 expense"。

#### 宪章修订(US1,BLOCKING)

- **FR-C001**: 宪章 MUST 从 v3.1.0 升至 v3.2.0 (MAJOR),原则一移除"转账、预算"的禁止项并保留"投资/OCR/导入导出/AI"仍属范围外;原则三允许 `Budget`/`Asset`/`Debt` 作为 `Family` 聚合内新实体。
- **FR-C002**: 宪章修订 MUST 附针对现有代码(027 前仅 income/expense、无 budget/asset 表)的迁移说明,满足治理章节"迁移计划"要求。

### Key Entities *(include if feature involves data)*

- **Transaction(扩展)**: 新增 `transfer` 类型;转账需关联转出账户与转入账户(现有 schema 仅有单一 `accountId`,需评估是否新增 `toAccountId` 或用双记录建模)。金额符号语义:income +/expense −/transfer 不参与收支聚合。
- **Budget(新增)**: 属于 `Family` 聚合内实体,**仅月预算**。`Budget(familyId, year, month, amount)`,(familyId, year, month) 唯一,year+month NOT NULL。amount 为分。统计页"年"周期不消费预算数据。年预算若 V2 需要再加 `periodType` 字段。
- **Account(扩展)**: 新增 `type` 字段(`asset` 资产 / `debt` 负债),显式区分账户类型。净资产/总资产/总负债按 type 分组聚合:`总资产 = SUM(asset 账户余额)`、`总负债 = SUM(debt 账户余额)`、`净资产 = 总资产 - 总负债`。账户余额仍由 `initialBalance + SUM(该账户 transactions.amount)` 推导。不新增 `Asset`/`Debt` 表(符合 YAGNI)。
- **Family(聚合根)**: 不变,Budget/Asset 仍在其内,通过 `familyId` 引用。

## 宪法修订提案 (v3.1.0 → v3.2.0 MAJOR)

为满足宪章治理章节对 MAJOR 修订的三项要求,本节即书面提案;迁移计划见各 User Story 的"独立测试"与 Key Entities。

### 原则一(MVP 范围)修订

- **现状(v3.1.0)**: "未经 PRD 修订,禁止新增范围外功能 —— 转账、预算、AI、OCR、导入导出、投资、多币种等。"
- **修订后(v3.2.0)**: "未经 PRD 修订,禁止新增范围外功能 —— **AI、OCR、导入导出、投资、多币种**等。**转账、预算、资产聚合、退款**自 spec 027-mobile-home-revamp 解锁,纳入 MVP 范围。"
- **理由**: 手机端首页设计(2026-07-14)把转账/预算/资产/退款列为首页核心模块;宪章作为"最高权威"应反映已确认的产品方向,而非用陈旧禁止项阻塞。

### 原则三(领域驱动)修订

- **现状(v3.1.0)**: "新表(`Asset`、`Debt`、`Budget`、`Investment`)在路线图解锁前属于范围外。"
- **修订后(v3.2.0)**: "`Budget` 自 027 解锁,作为 `Family` 聚合内实体(按月)。`Asset`/`Debt` 优先用现有 `Account.initialBalance` + 交易流水聚合推导,不新增表;仅当推导无法满足(如需独立负债建模)时再评估新增。`Investment` 仍属范围外。"
- **理由**: 资产概览只需净资产/总资产/总负债,可由现有账户+交易推导(YAGNI,原则六);预算必须新增 `Budget` 表(无法推导)。

### 版本号

v3.1.0 → v3.2.0 (MAJOR: 删除/重定义原则一禁止项清单 + 原则三新表清单;符合治理章节"MAJOR: 删除或根本性重定义某条原则")。

### 合规审查预期

`/speckit-plan` 的宪章检查在本 spec 上应输出:原则一(修订后)✅、原则三(修订后)✅、原则七(UI 调整纪律)⚠️ 提醒 —— 任何触及 `src/components/**/*.tsx`、`src/app/**/*.tsx` 的 JSX/className 改动必须先 `/heroui-react` skill 查询。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户打开首页后,无需下滑即可同时看到本月支出、收入、结余三项核心数据(首屏可见性)。
- **SC-002**: 用户切换月份后,首页所有受月份影响的模块在统一一次刷新内同步更新(无模块使用滞后月份)。
- **SC-003**: 320px 宽屏手机上,首页及明细/统计/记一笔/设置五屏均不出现横向滚动、内容重叠或文字截断。
- **SC-004**: 隐私模式开启时,应用内(含趋势图刻度/Tooltip/点击金额)无任何真实金额以任何形式可见。
- **SC-005**: 新用户(零交易/零账户/零预算)打开首页,不会看到任何空图表或空进度条,且能通过单一"记第一笔"按钮开始记账。
- **SC-006**: 转账记录在月度收支统计中完全不出现于收入或支出合计(转账隔离 100%)。
- **SC-007**: 预算使用达到 80% 与超过 100% 时,首页对应显示"接近超支"与"已超支"状态,无遗漏。
- **SC-008**: 单个首页模块加载失败时,其余模块仍可正常交互使用(模块隔离)。

## Assumptions

- 用户有稳定的移动端网络连接(与现有 MVP 假设一致)。
- 现有 026 的 HeroUI v3 栈、tRPC + Drizzle + PostgreSQL 架构、Better-Auth 认证不变;本次为 feature 纵切片,不重构框架。
- 隐私模式现有机制(localStorage + `<html>.privacy-on` + `.privacy-on [data-amount]` CSS)保留并扩展,不重写。

- 转账建模采用单一 `accountId`(转出)+ 新增 `toAccountId`(转入)的方向,而非双记录;最终由 plan 阶段在数据建模时定夺。

- 本次解锁范围(转账/预算/资产/退款)仅限本 spec;投资/OCR/导入导出/AI 仍属路线图,不在本 spec 内。
- 依赖现有系统:Better-Auth 会话、tRPC `protectedProcedure`、`Family` 聚合的 `familyId` 隔离机制。
