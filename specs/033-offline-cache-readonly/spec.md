# Feature Specification: 离线可读 + 写入队列同步(B 级离线)

**Feature Branch**: `033-offline-cache-readonly`

**Created**: 2026-07-18

**Status**: Draft

**Input**: User description + 3 项方向决策:
> 把 032 P4(离线数据缓存)立独立 spec。**B 级**:断网时能看 Dashboard 摘要 + 近期交易列表(只读),断网记账入队列、联网后台静默同步。不做 C 级完整双写/冲突解决。
>
> - Q1 缓存范围:**核心摘要 + 近期交易**(30/90 天,非全量历史)
> - Q2 写入同步:**后台静默同步**(Background Sync API,iOS 降级)
> - Q3 数据策略:**network-first**(缓存仅在断网时兜底,弱网仍服务器优先)

## Clarifications

### Session 2026-07-18

- **Q1 — 断网时要能看哪些数据?(缓存范围 + 空间)**
  → A:**核心摘要 + 近期交易**。缓存 Dashboard summary(本月收支/预算进度等聚合数字)+ 最近一段时间(默认 30 天,可调到 90 天)的交易列表明细。**不缓存全量历史**(几百上千条交易会占几十 MB,首同步慢、空间失控)。覆盖 90% 离线场景(用户想看"最近花了多少 / 这个月预算还剩多少 / 昨天那笔"),极少有人离线翻半年前的账。

- **Q2 — 断网记账后怎么同步到服务器?**
  → A:**后台静默同步**。断网记一笔 → 写入本地待同步队列(与 031 的 draft-storage 区分:那是"未提交草稿",这是"已确认提交但因断网未达服务器"的待发队列)→ 联网后浏览器 Background Sync API 自动触发后台重试提交,**用户无需手动操作**。失败才提醒。
  - **iOS 降级**:iOS Safari 不支持 Background Sync API,降级为"app 下次前台时检查队列并重试"。作为已知限制,不阻塞。

- **Q3 — 缓存数据与服务器数据冲突时谁优先?(读取策略)**
  → A:**network-first**。app 打开 / 联网时**总是先请求服务器**,服务器有数据就用最新的;**只有服务器请求失败(断网/超时)才回退到缓存**。
  - **不显示陈旧数据**(除非完全断网):弱网下仍是服务器数据优先,用户不会看到"余额是昨天的"这种误导。
  - **不需要"数据可能陈旧"UI 标记**:因为缓存只在断网兜底时出现,用户体感是"断网了所以看的是上次的数据"——这是合理预期,不需额外警告。

## User Scenarios & Testing *(mandatory)*

### User Story 0 - 幂等去重基石(clientRequestId)(Priority: P1,前置)

这是一个**横切基础设施故事**(非终端用户直接感知),作为 US2(断网记账队列同步)的
**前置基石**。Background Sync 的 at-least-once 投递语义 + SW retry / 前台 flush 竞态,
必然导致同一笔交易被提交多次。若无服务器端幂等,重复提交 = 重复记账 = Dashboard 余额
错——财务数据最不能容忍的失败。因此本 US **必须先于 US2 实现**(research R3/R4 耦合),
为整个离线写入路径提供"重复提交不重复记账"的硬保证。

用户视角(间接):用户断网记了一笔,联网后即使浏览器因网络抖动重试了多次,流水里该笔
交易**只出现一次**,余额正确。用户感知不到去重发生,但依赖它保证账目准确。

**Why this priority**:与 US1/US2 并列 P1,但**实现顺序最前**(无 UI、纯服务器 + migration,
可独立发 patch)。它是 US2 安全性的前提——没有它,US2 的后台同步会重复记账。

**Independent Test**:同一 `clientRequestId` 两次提交 `transaction.create` → 第二次返回与
第一次相同的 transaction id,DB 中该家庭仅 +1 行(非 +2);并发提交漏过 SELECT 检查时,
唯一索引兜底,最终仍仅 1 行。

**Acceptance Scenarios**:

1. **Given** 客户端带 `clientRequestId` 提交一笔交易,
   **When** 服务器处理,
   **Then** 正常创建,记录该 `clientRequestId`。
2. **Given** 同一 `clientRequestId` 再次提交(retry 场景),
   **When** 服务器处理,
   **Then** **不新建**,返回第一次创建的既有 transaction(相同响应,不报错——retry 幂等)。
3. **Given** 两个并发提交带同一 `clientRequestId`(漏过 SELECT 检查),
   **When** 同时 INSERT,
   **Then** 唯一索引兜底,第二个失败 → 回退返回既有;最终 DB 仅 1 行。
4. **Given** 提交不带 `clientRequestId`(向后兼容,如既有 API Key `/api/v1/` 路径),
   **When** 服务器处理,
   **Then** 走原逻辑正常创建(clientRequestId 字段为 NULL),既有路径不破。

### User Story 1 - 断网时仍能查看 Dashboard 与近期交易 (Priority: P1)

一位用户在地铁/飞机上打开 BALTHASAR(无网络)。与修复前(任何页面都打不开,只能看
offline.html 提示页)不同,用户现在能看到:Dashboard 的本月收支概览、预算进度、以及最近
30 天的交易列表明细。数据是上次联网时缓存的,虽然不是实时(完全断网),但足够让用户
回顾"最近花了多少 / 这个月还剩多少预算 / 昨天那笔午餐"。用户清楚地知道自己在断网兜底
模式,联网后会自动刷新到最新。

**Why this priority**:这是"离线可读"的核心价值,直接解决 032 P4 排除的"断网完全不可用"
问题。出差/通勤/旅行是记账 app 的高频使用场景,断网不可用 = 丢一次记账机会 = 可能忘掉。
P1 是整个 feature 的主体。

**Independent Test**:在 Chrome DevTools Application → Service Workers 勾"Offline",或手机
开飞行模式,打开 Dashboard 与流水页,确认能看到缓存的本月摘要 + 近期交易列表;联网后
确认数据自动刷新到最新。

**Acceptance Scenarios**:

1. **Given** 用户曾在联网时打开过 Dashboard 与流水页(缓存已写入),
   **When** 用户断网后再次打开 Dashboard,
   **Then** 用户看到上次缓存的本月收支摘要与预算进度(不是空白/offline.html),页面顶部有"离线模式"轻量提示。
2. **Given** 同上断网状态,
   **When** 用户打开流水页,
   **Then** 用户看到最近 30 天(默认)的交易列表明细,可滚动浏览、按日分组查看。
3. **Given** 断网时查看的缓存数据,
   **When** 用户恢复联网,
   **Then** Dashboard 与流水自动重新请求服务器,数据刷新到最新(无需手动下拉)。
4. **Given** 用户从未联网打开过(无缓存)且当前断网,
   **When** 打开 Dashboard,
   **Then** 显示 offline.html 或明确的"无缓存数据,请联网后首次加载"提示(不显示空白/错误)。
5. **Given** 缓存的交易列表,
   **When** 用户在断网时尝试查看缓存范围之外(如 90 天前)的交易,
   **Then** 系统明确提示"此条目需联网查看"(不假装有数据)。

### User Story 2 - 断网记账入队列,联网后台静默同步 (Priority: P1)

一位用户在断网状态下记一笔(地铁里刚吃完午餐)。用户填完表单点"确认记账",app **不报错**,
而是把这笔交易放入本地"待同步队列",给用户一个轻量提示("已记录,联网后自动同步")。
用户继续做别的事。当用户手机恢复网络后,浏览器在后台自动把队列里的交易提交到服务器,
**用户无需重新打开 app、无需手动点"同步"**。提交成功后,这笔交易出现在流水里,就像
正常记账一样。只有提交失败(如服务器持续不可达)才提醒用户。

**Why this priority**:与 US1 并列 P1——"离线可读"解决看,"离线可写"解决记。两者合起来
才是完整的"断网可用"。直接服务宪章五"10 秒记账":不能让断网打断记账节奏。后台静默同步
是体验最顺滑的方案(Q2 决策)。

**Independent Test**:DevTools Offline 模式下记一笔 → 确认入队列 + 提示 → 取消 Offline →
等待后台同步触发 → 确认交易出现在流水 + 队列清空。

**Acceptance Scenarios**:

1. **Given** 用户断网,
   **When** 用户填完表单点"确认记账",
   **Then** 交易进入本地待同步队列,UI 显示轻量提示(如"已记录,联网后自动同步"),不显示错误。
2. **Given** 队列中有待同步交易,
   **When** 浏览器/网络恢复(Background Sync 触发,或 app 下次前台),
   **Then** 系统自动把队列交易逐条提交到服务器,无需用户操作。
3. **Given** 后台同步成功,
   **When** 用户下次打开流水页,
   **Then** 该笔交易已出现在列表中(与正常记账无异),队列中该条清除。
4. **Given** 后台同步某条失败(如服务器 5xx),
   **When** 重试次数达上限仍失败,
   **Then** 系统提醒用户"有 1 笔交易未能同步,点击查看",提供手动重试或丢弃选项。
5. **Given** 用户在断网时记了多笔(队列多条),
   **When** 联网后台同步,
   **Then** 按用户记账的时间顺序逐条提交(避免乱序),全部成功后队列清空。
6. **Given** iOS 用户(iOS 不支持 Background Sync API),
   **When** 断网记账后恢复网络,
   **Then** 下次打开 app 时自动检查并重试队列(降级方案,作为已知限制,不阻塞)。

### User Story 3 - 弱网/正常场景不被缓存拖累(network-first) (Priority: P2)

一位用户在正常网络下使用 BALTHASAR。虽然现在有了离线缓存,但用户**感觉不到缓存的
存在**:每次打开 Dashboard,看到的总是服务器最新数据(不是缓存的旧数据);弱网下可能
稍慢等服务器响应,但绝不会显示昨天的余额。缓存只在真正断网时才作为兜底出现。

**Why this priority**:这是 Q3 决策的直接体现——network-first 保证数据新鲜度。P2 是因为它
是"约束"而非"新能力"(P1 的 US1/US2 是新能力),但它是整个 feature 可信度的基石:
如果缓存机制让用户看到陈旧余额,比"断网不可用"更糟(财务数据误导)。

**Independent Test**:在正常网络下打开 Dashboard,确认显示的是服务器最新数据(可通过
服务器改一笔后刷新验证);断开网络再打开,确认回退到缓存;两者数据一致或缓存明确标注
是上次的数据。

**Acceptance Scenarios**:

1. **Given** 用户在正常网络下,
   **When** 打开 Dashboard,
   **Then** 显示服务器最新数据(不是缓存),即使缓存存在。
2. **Given** 弱网(服务器响应慢但可达),
   **When** 打开 Dashboard,
   **Then** 等待服务器响应(显示加载态),不立即显示缓存旧数据。
3. **Given** 服务器请求失败(断网/超时),
   **When** 同上打开,
   **Then** 回退到缓存数据,页面顶部有"离线模式"轻量提示。
4. **Given** 缓存数据存在,
   **When** 用户查看,
   **Then** 不需要额外的"数据可能陈旧"警告 UI(因为缓存仅在断网兜底时出现,用户预期就是旧数据)。

### User Story 4 - 缓存空间与新鲜度可管理 (Priority: P3)

随着用户长期使用,缓存的数据量增长。系统自动管理缓存空间:只保留最近一段时间(默认
30 天,可配置到 90 天)的交易 + 当前月份的 Dashboard 摘要;过旧的数据自动清理。用户
无需手动清缓存,app 也不会无限膨胀占用设备空间。

**Why this priority**:P3,因为是"长期可用性"而非"核心功能"。但不做会导致缓存几个月后
膨胀到几十 MB,违反移动端空间预算。作为工程兜底,与 P1/P2 同期交付。

**Independent Test**:查看应用存储占用(IDB 大小),确认长期使用后稳定在合理上限(如
< 10MB);确认 30 天前的交易不在缓存里(需联网查看)。

**Acceptance Scenarios**:

1. **Given** 用户长期使用(累计大量交易),
   **When** 查看应用缓存占用,
   **Then** 缓存大小稳定在合理上限(建议 < 10MB),不随历史无限增长。
2. **Given** 缓存中有超过保留期(默认 30 天)的交易,
   **When** 系统执行缓存清理,
   **Then** 过旧交易从缓存移除(服务器仍有,只是离线不可见)。
3. **Given** 当前月份的 Dashboard 摘要,
   **When** 跨月(进入下个月),
   **Then** 缓存更新为新月份数据,上月摘要按需保留或清理(由实现决定,不阻塞)。

### Edge Cases

- **首次使用即断网**:用户首次安装、从未联网打开过 → 无缓存。断网打开 Dashboard 应显示
  offline.html 或"无缓存,请联网首次加载",**不显示空白或崩溃**。
- **登录态失效**:断网期间 session 过期 → 后台同步提交会收到 401。系统应区分"网络失败"
  (重试)与"认证失效"(停止重试,提示重新登录),不无限重试 401。
- **队列冲突**:用户断网记了一笔,联网同步前又在另一设备记了同分类/同时段交易 → 不做
  冲突合并(C 级才做),两笔都保留(它们是独立的交易,时间戳不同)。仅当服务器返回
  业务错误(如账户不存在了)才提示用户。
- **部分同步成功**:队列 3 条,前 2 条成功第 3 条失败 → 前 2 条出队,第 3 条保留并重试,
  不回滚已成功的。
- **缓存损坏**:IDB 数据损坏/无法读取 → 优雅降级为"无缓存"行为(回退到 network-first
  的失败路径),不崩溃;可清理重建。
- **多标签页/多设备并发**:用户在两个标签页同时操作 → 队列与缓存应是同一 IDB,浏览器
  自动处理 IDB 事务;不出现重复提交(每条队列项有客户端生成的唯一 ID,服务器去重或
  幂等处理)。
- **Background Sync 不触发的浏览器**:除 iOS 外,部分旧版浏览器/桌面 Firefox 可能不支持
  Background Sync → 降级为"app 前台时检查队列"(同 iOS 降级路径)。
- **隐私模式**:无痕模式下 IDB 可能受限或清空 → 不阻塞正常浏览(退化为非离线模式),
  不报错。
- **缓存与服务器 schema 演进**:未来服务器改了交易字段结构,旧缓存可能不兼容 → 缓存
  带版本号,版本不匹配时丢弃重建(不尝试迁移,简化)。
- **离线编辑/删除**:本 feature **只做离线创建(create)**,不做离线编辑/删除(B 级范围)。
  断网时编辑/删除交易应禁用或提示"需联网操作"(避免复杂的离线删除一致性)。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在本地(浏览器持久化存储)缓存最近一段时间(默认 30 天,可配置
  到 90 天)的交易列表明细 + 当前月份的 Dashboard 摘要(本月收支/预算进度等聚合数字)。
- **FR-002**: 断网时(服务器请求失败),Dashboard 与流水页 MUST 回退显示缓存数据,而非
  空白或 offline.html;页面 MUST 有"离线模式"轻量提示。
- **FR-003**: 正常/弱网下,系统 MUST 优先请求服务器最新数据(network-first);**只有**
  服务器请求失败才用缓存。缓存不得在服务器可达时被优先显示。
- **FR-004**: 断网时记一笔,系统 MUST 把交易写入本地"待同步队列"(区别于 031 的未提交
  草稿 draft-storage),并给用户轻量提示("已记录,联网后自动同步"),不报错。
- **FR-005**: 网络/浏览器恢复时,系统 MUST 自动后台重试提交队列中的交易(Background Sync
  API),无需用户手动操作。
- **FR-006**: 后台同步 MUST 按用户记账时间顺序逐条提交;部分成功时已成功的出队、失败的
  保留重试,不回滚已成功项。
- **FR-007**: 同步失败的交易(重试达上限或服务器业务错误)MUST 提醒用户,提供手动重试
  或丢弃选项;**不无限重试**(尤其 401 认证失效要停止并提示重新登录)。
- **FR-008**: 队列中每条交易 MUST 有客户端生成的唯一 ID,确保重复提交时服务器可去重
  (幂等),避免多标签页/重试导致重复记账。
- **FR-009**: iOS Safari(不支持 Background Sync API)MUST 降级为"app 前台时检查并重试
  队列",作为已知限制记录,不阻塞核心功能。
- **FR-010**: 系统 MUST 自动管理缓存空间:只保留保留期内的交易 + 当前月摘要,过旧数据
  自动清理;缓存大小长期稳定在合理上限(建议 < 10MB)。
- **FR-011**: 从未联网打开过(无缓存)且断网时,MUST 显示 offline.html 或明确的"无缓存,
  请联网首次加载"提示,不显示空白/错误。
- **FR-012**: 缓存范围之外(如 90 天前)的交易,断网时 MUST 明确提示"需联网查看",不
  假装有数据。
- **FR-013**: 本 feature **只做离线创建**;断网时的编辑/删除交易 MUST 禁用或提示"需联网
  操作"(不做离线编辑/删除一致性,B 级范围限定)。
- **FR-014**: 缓存 MUST 带版本号;未来服务器 schema 演进导致不兼容时,缓存 MUST 丢弃
  重建而非尝试迁移。
- **FR-015**: 所有缓存读写、队列操作 MUST 不破坏既有 PWA 功能(SW 注册/离线 offline.html/
  安装引导/更新流程/031 草稿 draft-storage)—— 回归 0 缺陷。
- **FR-016**: 隐私敏感字段的处理 MUST 与既有隐私模式(privacy-lock 等)一致;缓存中的
  交易数据在隐私锁定时同样不可见(不绕过隐私模式)。

### Key Entities

本 feature 新增**客户端持久化实体**(非服务器 DB,存在浏览器 IndexedDB):

- **缓存交易(CachedTransaction)**:近期交易的本地副本,含服务器返回的交易字段 + 缓存
  写入时间戳。关系:属于一个家庭(familyId),按 occurredAt 排序。保留期内有效。
- **缓存摘要(CachedDashboardSummary)**:当前月份 Dashboard 聚合数据的本地副本(本月
  收入/支出/预算进度等)。关系:属于一个家庭 + 年月。新月度时更新。
- **待同步队列项(PendingTransaction)**:断网记账产生的待提交交易,含客户端生成的唯一
  ID + 完整表单数据 + 创建时间戳 + 重试计数。关系:独立项,按创建顺序排队。
- **缓存元数据(CacheMeta)**:缓存版本号、最后同步时间、保留期配置等。

不新增服务器端实体(复用既有 Transaction 表 + dashboard summary procedure)。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 断网时打开 Dashboard,用户在 **2 秒内**看到缓存的本月摘要 + 预算进度(不
  是空白/offline.html),以真机/DevTools Offline 走查确认(定性 + 时长)。
- **SC-002**: 断网时打开流水页,用户能看到最近 **30 天**(默认)的交易列表,可滚动浏览、
  按日分组,加载无明显卡顿(< 1 秒)。
- **SC-003**: 断网记一笔,从点"确认记账"到入队列 + 提示,**< 1 秒**(不卡顿,符合宪章五
  "10 秒记账"预算)。
- **SC-004**: 联网恢复后,后台同步在 **30 秒内**自动完成(用户无感),交易出现在流水;
  无需用户手动操作。
- **SC-005**: 正常网络下,Dashboard 显示的**总是**服务器最新数据(network-first 验证):
  服务器改一笔后刷新,用户立即看到更新,不被缓存挡住。
- **SC-006**: 长期使用(累计 100+ 交易)后,应用缓存占用**稳定 < 10MB**,不随历史无限
  增长(机械化测量 IDB 大小)。
- **SC-007**: 既有 PWA 功能(SW/离线 offline.html/安装/更新/031 草稿)回归 **0 缺陷**。
- **SC-008**: 桌面端回归 **0 缺陷**(桌面无虚拟键盘但有网络切换/离线场景,需同样可用)。

## Assumptions

- **沿用既有环境基线**:PWA / 移动浏览器(iOS Safari、Android Chrome)+ 桌面 Chrome/Edge。
  Background Sync API 以 Android Chrome + 桌面 Chrome 为主要验收平台;iOS Safari 降级为
  前台重试(已知限制)。
- **沿用 031/032 的 PWA 基础设施**:既有 SW(`public/sw.js`)+ `useVisualViewport` 等
  hooks 不动;本 feature 在其之上加数据缓存层,不替换 SW 本身。
- **保留期默认 30 天**:可配置(90 天上限),由实现阶段定具体取值;不暴露给用户调
  (YAGNI,内部参数)。
- **空间预算 < 10MB**:基于"30 天交易 × 平均 200 字节/条 + 摘要"估算,家庭记账场景下
  足够宽裕。实现阶段实测确认。
- **不做 C 级双写/冲突解决**:离线编辑/删除不在范围(FR-013);多设备并发记账不做合并
  (Edge Case:两笔独立保留)。若未来需要,单独立 C 级 spec。
- **复用既有 tRPC procedure**:不新增服务器端点;缓存层在客户端拦截/回退 tRPC query,
  写入队列复用既有 `transaction.create` procedure 提交。
- **与 031 draft-storage 区分**:draft-storage 是"用户未点提交的草稿"(防意外丢失),
  本 feature 的 PendingTransaction 队列是"用户已点提交但因断网未达服务器"(待发队列)。
  两者独立,不合并(语义不同)。
- **缓存版本号策略**:简单递增整数,不兼容时丢弃重建(不迁移),简化实现(YAGNI)。
- **依赖宪章原则七(UI 调整纪律)**:本 feature 触及"离线模式"UI 提示 + 可能的"待同步"
  徽标,涉及 `src/components/**/*.tsx` className/JSX,MUST 先查 `/heroui-react` skill。
- **iOS 限制作为已知**:Background Sync API 在 iOS Safari 不支持,降级方案为前台重试;
  iOS 下断网记账仍可用(入队列),只是同步时机延后到下次打开 app。
